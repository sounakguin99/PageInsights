2. CRITICAL ARCHITECTURE DECISION — USE GOOGLE PSI API DIRECTLY
═══════════════════════════════════════════════════════════════════════════

⚠️ KEY CHANGE FROM ORIGINAL SPEC:

Instead of running Lighthouse locally via subprocess (fragile, slow, 
resource-heavy), we call the official Google PageSpeed Insights API v5 
directly from our Node.js (Express) backend.

API Endpoint:
  GET https://www.googleapis.com/pagespeedonline/v5/runPagespeed

Required Parameters:
  • url           — The target URL to analyze
  • strategy      — "mobile" OR "desktop" (we run BOTH, separately)
  • category      — Must pass ALL 4:
                     - performance
                     - seo
                     - accessibility
                     - best-practices

Full request example:
  GET https://www.googleapis.com/pagespeedonline/v5/runPagespeed
    ?url=https://www.sounakguin.in/
    &strategy=mobile
    &category=performance
    &category=seo
    &category=accessibility
    &category=best-practices

This returns the EXACT SAME data Google shows on pagespeed.web.dev.

WHY THIS IS BETTER:
  ✅ 100% score accuracy — identical to Google's own results
  ✅ No local Lighthouse/Chrome install needed
  ✅ No heavy subprocess management
  ✅ No CPU throttling simulation issues
  ✅ Free tier allows ~25,000 queries/day (with API key)
  ✅ Returns both Lab Data AND Field Data (CrUX)

5. MOBILE vs DESKTOP — KEY DIFFERENCES
═══════════════════════════════════════════════════════════════════════════

Google runs TWO completely separate Lighthouse analyses:

┌────────────────────────┬─────────────────────────────────────────────┐
│ Setting                │ Mobile                  │ Desktop           │
├────────────────────────┼─────────────────────────┼───────────────────┤
│ strategy param         │ "mobile"                │ "desktop"         │
│ CPU Throttling         │ 4x slowdown (Moto G4)   │ No throttling     │
│ Network Throttling     │ Slow 4G (1.6 Mbps)      │ No throttling     │
│ Screen Size            │ 412×823 px              │ 1350×940 px       │
│ User Agent             │ Mobile Chrome           │ Desktop Chrome    │
│ Scores                 │ Usually LOWER           │ Usually HIGHER    │
│ Device-specific audits │ tap-targets, font-size  │ Not applicable    │
└────────────────────────┴─────────────────────────┴───────────────────┘

⚠️ OUR APP MUST:
  1. Run BOTH mobile AND desktop analyses per URL
  2. Store both results separately in DB
  3. Let user toggle between Mobile/Desktop on the dashboard
  4. Show DIFFERENT scores, metrics, issues for each strategy

7. BACKEND IMPLEMENTATION
═══════════════════════════════════════════════════════════════════════════

Step 1: Express Endpoints
─────────────────────────
  POST /analyze         — Accept { url: string }
                          → Create project in DB
                          → Trigger async background job
                          → Return { job_id }

  GET /status/:id       — Return { status, progress }

  GET /report/:id       — Return full report + audits for BOTH strategies
                          → { mobile: {...}, desktop: {...} }

  GET /report/:id?strategy=mobile   — Return single strategy report

Step 2: Background Worker Flow
──────────────────────────────
  run_scan(project_id, url):
    1. Update project status → "processing"
    
    2. Call PSI API for MOBILE:
       GET https://www.googleapis.com/pagespeedonline/v5/runPagespeed
         ?url={url}
         &strategy=mobile
         &category=performance
         &category=seo
         &category=accessibility
         &category=best-practices
    
    3. Call PSI API for DESKTOP:
       Same URL but &strategy=desktop
    
    4. Parse BOTH responses:
       - Extract category scores (× 100)
       - Extract all 6 performance metrics
       - Extract field data (CrUX) if available
       - Extract all audits by category
       - Classify audits into: opportunity/diagnostic/passed/not-applicable
       - Extract screenshots and filmstrip
       - Calculate severity for each failing audit
    
    5. Save to Supabase:
       - Insert report row (mobile)
       - Insert report row (desktop)
       - Insert all individual audit rows
       - Insert simplified issues for quick display
       - Upload screenshots to Supabase Storage
    
    6. Update project status → "done"

Step 3: PSI API Response Parser
───────────────────────────────
  The parser MUST handle:
  
  a) lighthouseResult.categories.performance.score → × 100
  b) lighthouseResult.categories.accessibility.score → × 100
  c) lighthouseResult.categories["best-practices"].score → × 100
  d) lighthouseResult.categories.seo.score → × 100
  
  e) For each audit in lighthouseResult.audits:
     - Read the category it belongs to via categories.X.auditRefs
     - Determine its group via auditRefs[].group:
       • "metrics" → metric
       • "load-opportunities" → opportunity
       • "diagnostics" → diagnostic
       • (no group + score=1) → passed
       • (scoreDisplayMode="notApplicable") → not-applicable
  
  f) For field data:
     - loadingExperience.metrics.FIRST_CONTENTFUL_PAINT_MS.percentile
     - loadingExperience.metrics.LARGEST_CONTENTFUL_PAINT_MS.percentile
     - loadingExperience.metrics.INTERACTION_TO_NEXT_PAINT.percentile
     - loadingExperience.metrics.CUMULATIVE_LAYOUT_SHIFT_SCORE.percentile
     - loadingExperience.metrics.EXPERIMENTAL_TIME_TO_FIRST_BYTE.percentile
     - loadingExperience.overall_category
  
  g) For screenshots:
     - lighthouseResult.audits["final-screenshot"].details.data
     - lighthouseResult.audits["screenshot-thumbnails"].details.items[]

Step 4: Score Calculation
─────────────────────────
  Google's official Lighthouse scoring:
  
  Performance score is a weighted median of:
    FCP  → 10%
    SI   → 10%
    LCP  → 25%
    TBT  → 30%
    CLS  → 25%
  
  We DON'T recalculate this — we use the score Google returns directly.
  
  Our "Overall Score" (custom composite for our UI):
    overall = (
      performance × 0.4 +
      seo × 0.3 +
      accessibility × 0.2 +
      best_practices × 0.1
    )
